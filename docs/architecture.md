# 架构与导入协议

本文描述 Codex Dream Skin Store 的目标边界、静态 MVP 结构，以及商店与本机客户端之间的 `dreamskin://` v0 契约。除非明确写为“当前已有”，其余内容均是实现约束或路线图，不代表客户端已经支持。

## 1. 系统边界

```text
┌──────────────────────────────────────────┐
│ Store（低权限 Web 应用）                 │
│ 目录 / 搜索 / 详情 / 下载元数据 / 审核   │
└───────────────────┬──────────────────────┘
                    │ HTTPS 包地址 + 摘要
                    │ dreamskin:// 用户手势
                    ▼
┌──────────────────────────────────────────┐
│ Dream Skin Client（本机信任边界）        │
│ 确认 / 下载 / 校验 / 解包 / 安装 / 回滚  │
└───────────────────┬──────────────────────┘
                    │ 已验证的声明式主题
                    ▼
┌──────────────────────────────────────────┐
│ Codex Desktop + loopback-only CDP        │
│ 读取 theme.json / 图片并应用视觉效果     │
└──────────────────────────────────────────┘
```

商店不承担以下职责：

- 不打开或代理本机 CDP；
- 不直接写入用户的主题目录；
- 不修改 Codex Desktop、`app.asar` 或应用签名；
- 不读取 Codex 凭证、项目内容和会话数据；
- 不把网页中的“已点击”解释为“客户端已安装成功”。

该项目不是 Codex 插件。未来可以另做主题制作辅助插件，但它不能替代负责系统协议、文件校验和本机安装的 Dream Skin 客户端。

## 2. 当前客户端能力与缺口

当前 [Codex Dream Skin](https://github.com/Fei-Away/Codex-Dream-Skin) 的 macOS 实现已经：

- 使用本地 `themes/<id>/` 存放主题；
- 读取 `theme.json` 和同目录图片；
- 能按主题 ID 切换，并在 CDP 可用时热重载。

商店导入仍依赖客户端补齐：

1. 注册 `dreamskin` 系统协议及 `.dreamskin` 文件关联；
2. 提供接受本地文件或 HTTPS URL 的安装入口；
3. 在临时目录下载、限额、校验、解包和验证主题；
4. 使用原子目录替换安装到 `themes/<id>/`，冲突时保留可恢复备份；
5. 将安装与“立即应用”作为两个可确认动作；
6. 为 Windows 提供与 macOS 一致的声明式主题和切换语义。

在这些能力完成前，商店只能展示或复制导入链接，并提供手动下载说明。

## 3. 静态 MVP 架构

MVP 的读取路径应保持简单：

```text
版本化主题元数据 ──> 服务端/构建时读取 ──> 列表与详情页
       │                                        │
       └──── HTTPS package URL + SHA-256 ──────┘
                                                │
                                                └─> 导入链接
```

静态目录中的每个发布版本至少需要这些语义字段：

```json
{
  "schemaVersion": 1,
  "id": "author.theme-id",
  "name": "Theme Name",
  "version": "1.0.0",
  "summary": "简短、可审核的主题说明",
  "author": {
    "name": "Author"
  },
  "preview": {
    "url": "https://cdn.example.com/theme/preview.webp"
  },
  "package": {
    "url": "https://cdn.example.com/theme/1.0.0.dreamskin",
    "sha256": "<64-lowercase-hex>",
    "size": 1234567
  },
  "compatibility": {
    "client": ">=1.0.0",
    "platforms": ["macos"]
  },
  "tags": ["light", "illustration"]
}
```

这是数据模型建议，不是对当前代码中字段名的承诺。落地 schema 后应进行结构校验，并让未知字段默认失效，而不是被透传到客户端执行。

静态 MVP 的写入路径是受审查的代码贡献：作者提交元数据和文件摘要，CI 验证结构、URL、重复 ID、版本顺序和摘要格式。在线上传、D1 与 R2 绑定可以在需要作者后台时再启用。

## 4. `.dreamskin` 包提案

`.dreamskin` 可以是具有固定结构的 ZIP 容器，但客户端必须按不可信输入处理。为贴合现有 macOS 主题目录，v0 建议把 `theme.json` 及其引用图片放在包根目录：

```text
manifest.json
theme.json
background.webp
preview.webp
signature.json       # 可选；签名阶段启用
```

约束如下：

- `manifest.json` 声明包 schema、主题 ID、版本、文件清单、每个文件的摘要和兼容范围；
- `theme.json` 只能使用客户端公开的字段白名单，图片引用必须是无路径分隔符的包内文件名；
- 包内不得包含 JS、HTML、脚本、二进制程序、符号链接或设备文件；
- 预览图不参与运行时渲染时，客户端可以不安装它；
- 清单 ID 与目录名不直接信任，客户端验证后生成安全目标路径；
- 未识别的 schema 版本应拒绝安装，而不是猜测兼容。

包的外部 SHA-256 记录在商店版本元数据和导入 URL 中。它用于发现 CDN 文件被替换或下载损坏。发布者签名用于证明来源，两者解决的问题不同，不能互相替代。

## 5. `dreamskin://` v0 契约

### 5.1 URI 形式

```text
dreamskin://install?url=<percent-encoded-https-url>&sha256=<64-lowercase-hex>&themeId=<optional>&version=<optional>
```

| 参数 | 要求 | 语义 |
| --- | --- | --- |
| `url` | 必填 | `.dreamskin` 的公开 HTTPS URL，必须完整进行百分号编码 |
| `sha256` | 必填 | 整个下载文件的 SHA-256，小写十六进制，共 64 位 |
| `themeId` | 可选 | 商店展示或诊断提示，不是安装目标路径的可信来源 |
| `version` | 可选 | 商店展示或诊断提示，最终版本以验证后的包清单为准 |

生产环境只接受 HTTPS。开发时是否允许 loopback HTTP 由客户端通过明确的开发模式控制，不能由 URL 参数开启。

### 5.2 商店行为

1. 只在用户点击“导入”时导航到自定义 URI；
2. 对嵌套 URL 使用标准 `URLSearchParams`/百分号编码，禁止字符串拼接；
3. 页面同时提供包大小、摘要、作者、版本、平台和手动下载入口；
4. 唤起后只显示“已发送到客户端”，除非未来有经过认证的回执机制；
5. 不循环唤起、不后台弹窗，也不尝试探测本机 CDP。

### 5.3 客户端行为

1. 严格解析 scheme 与 action，拒绝重复的必填参数和畸形编码；
2. 显示来源域名、主题 ID、版本、大小（若已知）并要求确认；
3. 限制重定向次数、最终协议、下载字节数和超时；
4. 将下载保存到新建的私有临时目录，先校验整个包的 SHA-256；
5. 安全解包并逐文件校验清单，拒绝绝对路径、`..`、替代数据流、符号链接和大小超限；
6. 校验 JSON schema、主题字段白名单、图片真实格式、像素尺寸及兼容版本；
7. 原子安装并保留失败回滚点；
8. 成功后再次询问是否立即切换，不把安装自动等同于应用。

URL 中的 `themeId` 与 `version` 仅用于在下载前展示。它们和包内信息不一致时必须中止，并向用户报告来源不一致。

## 6. 信任与审核模型

### 完整性

- 商店记录不可变版本 URL、字节大小和 SHA-256；
- 发布后改变包内容必须创建新版本，不得只替换同一 URL；
- 客户端以下载到的原始字节计算摘要。

### 发布者身份

- 后续为发布者密钥和清单签名定义可轮换的信任链；
- 签名覆盖规范化清单和全部运行时文件摘要；
- 密钥撤销与恶意版本撤回需要独立、可缓存的元数据。

### 内容安全

- 自动检查文件类型、解包行为、尺寸、字段白名单和 URL；
- 人工检查预览与实际主题一致性、版权、冒充和不当内容；
- 已发布主题仍可举报、下架或阻止特定版本继续安装；
- 主题许可证必须由作者显式声明，商店代码的 MIT 许可证不会自动覆盖主题作品。

## 7. 版本与兼容性

主题、包 schema、目录 schema 和客户端应各自版本化：

- 主题 `version` 使用 SemVer；
- `manifest.schemaVersion` 控制包解析规则；
- 目录 `schemaVersion` 控制商店元数据读取；
- `compatibility.client` 与 `platforms` 决定商店提示和客户端最终校验。

商店可以隐藏明显不兼容的导入按钮，但客户端永远是最终裁决者。旧客户端不认识新 schema 时必须安全失败。

## 8. 演进顺序

1. 固化目录 schema、样例包和校验测试；
2. 在客户端实现本地文件安装，再增加 `dreamskin://` URL 入口；
3. 用端到端样例验证商店链接、下载、哈希错误、兼容错误和回滚；
4. 引入签名、撤回清单与 Windows 对齐；
5. 确有在线写入需求后，再加入身份、D1、R2、上传和审核队列。

这条顺序把协议和安全边界放在平台规模化之前，也让静态 MVP 能持续作为故障时的只读目录。
