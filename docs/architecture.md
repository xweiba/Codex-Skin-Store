# 架构与导入协议

本文描述 Codex Dream Skin Store 的目标边界、静态 MVP 结构，以及商店与本机客户端之间的 `dreamskin://` MVP v1 契约。除非明确写为“当前已有”，其余内容均是实现约束或路线图，不代表客户端已经支持。

## 1. 系统边界

```text
┌──────────────────────────────────────────┐
│ Store（低权限 Web 应用）                 │
│ 目录 / 搜索 / 详情 / 下载元数据 / 审核   │
└───────────────────┬──────────────────────┘
                    │ HTTPS 包地址 + 摘要
                    │ dreamskin:// 用户手势
                    ▼
┌──────────────────────────────────────────┐
│ Dream Skin Client（本机信任边界）        │
│ 确认 / 下载 / 校验 / 解包 / 安装 / 回滚  │
└───────────────────┬──────────────────────┘
                    │ 已验证的声明式主题
                    ▼
┌──────────────────────────────────────────┐
│ Codex Desktop + loopback-only CDP        │
│ 读取 theme.json / 图片并应用视觉效果     │
└──────────────────────────────────────────┘
```

商店不承担以下职责：

- 不打开或代理本机 CDP；
- 不直接写入用户的主题目录；
- 不修改 Codex Desktop、`app.asar` 或应用签名；
- 不读取 Codex 凭证、项目内容和会话数据；
- 不把网页中的“已点击”解释为“客户端已安装成功”。

该项目不是 Codex 插件。未来可以另做主题制作辅助插件，但它不能替代负责系统协议、文件校验和本机安装的 Dream Skin 客户端。

## 2. 当前客户端能力与缺口

Windows 客户端已经支持本地 `.dreamskin`、`dreamskin://install`、系统文件关联、安全 HTTPS 下载、大小与 SHA-256 限制、Ed25519 验证、原子安装，以及“安装”和“立即应用”两次独立确认。

兼容的 macOS 实现代码已经准备，包括本地包导入、URL handler、同等下载限制、签名验证和安装/应用分离确认；但它尚未在真实 macOS GitHub runner 或设备完成验收。因此当前状态必须区分“代码已实现”和“平台已验证”，不能把后者提前标记为完成。

商店已经发布首个真实签名样例。只有静态目录中同时具备不可变 HTTPS URL、精确字节数、SHA-256、包 ID、版本和 `published: true` 的条目，才会生成一键导入与手动下载入口；预览条目使用 `package: null`。

## 3. 静态 MVP 架构

MVP 的读取路径应保持简单：

```text
版本化主题元数据 ──> 服务端/构建时读取 ──> 列表与详情页
       │                                        │
       └──── HTTPS package URL + SHA-256 ──────┘
                                                │
                                                └─> 导入链接
```

静态目录中的每个发布版本至少需要这些语义字段：

```json
{
  "schemaVersion": 1,
  "id": "author.theme-id",
  "name": "Theme Name",
  "version": "1.0.0",
  "summary": "简短、可审核的主题说明",
  "author": {
    "name": "Author"
  },
  "preview": {
    "url": "https://cdn.example.com/theme/preview.webp"
  },
  "package": {
    "url": "https://cdn.example.com/theme/1.0.0.dreamskin",
    "sha256": "<64-lowercase-hex>",
    "size": 1234567
  },
  "compatibility": {
    "client": ">=1.0.0",
    "platforms": ["macos"]
  },
  "tags": ["light", "illustration"]
}
```

正式目录 schema 位于 [`spec/store-theme.schema.json`](../spec/store-theme.schema.json)。每个主题在 `catalog/themes/` 中占一个与 slug 同名的 JSON 文件，未知字段默认拒绝；构建前由校验脚本检查结构、URL、摘要、大小和唯一性，再生成类型安全的静态目录。当前签名样例拥有真实 `package` 元数据，其余展示主题使用 `package: null`，不会生成导入入口。

静态 MVP 的写入路径是受审查的代码贡献：作者提交元数据和文件摘要，CI 验证结构、URL、重复 ID、版本顺序和摘要格式。在线上传、D1 与 R2 绑定可以在需要作者后台时再启用。

## 4. `.dreamskin` MVP v1 包

`.dreamskin` 是具有固定结构的 ZIP 容器，客户端必须始终按不可信输入处理。包根目录只允许恰好三个普通文件：

```text
theme.json
background.webp      # 也可以是 .png、.jpg 或 .jpeg
preview.webp         # 也可以是 .png、.jpg 或 .jpeg
```

约束如下：

- `theme.json` 同时承担运行时主题、包清单和签名清单的职责，必须通过严格 Schema 校验；
- `assets` 声明两张图片的路径、媒体类型、字节数、像素尺寸和 SHA-256，`image` 必须等于背景图路径；
- `signature` 必填，使用 Ed25519 对 RFC 8785 规范化清单签名，发布包的 `keyId` 必须在客户端信任密钥环中；
- 包内不得包含 JS、CSS、HTML、SVG、脚本、二进制程序、符号链接、设备文件或任何额外条目；
- 客户端先执行 ZIP 路径与大小限制，再校验 Schema、签名、资源摘要和图片真实格式；
- 清单 ID 与文件名都不直接信任，客户端验证后生成安全目标路径；
- 未识别的 `packageVersion` 或 `schemaVersion` 必须拒绝安装，而不是猜测兼容。

包的外部 SHA-256 记录在商店版本元数据和导入 URL 中。它用于发现 CDN 文件被替换或下载损坏。发布者签名用于证明来源，两者解决的问题不同，不能互相替代。

## 5. `dreamskin://` MVP v1 契约

### 5.1 URI 形式

```text
dreamskin://install?url=<percent-encoded-https-url>&sha256=<64-lowercase-hex>&size=<decimal>&id=<theme-id>&version=<semver>
```

| 参数 | 要求 | 语义 |
| --- | --- | --- |
| `url` | 必填 | `.dreamskin` 的公开 HTTPS URL，必须完整进行百分号编码 |
| `sha256` | 必填 | 整个下载文件的 SHA-256，小写十六进制，共 64 位 |
| `size` | 必填 | 商店声明的包字节数，范围为 1 到 20971520 |
| `id` | 可选 | 商店展示或诊断提示，不是安装目标路径的可信来源 |
| `version` | 可选 | 商店展示或诊断提示，最终版本以验证后的包清单为准 |

生产环境只接受 HTTPS。开发时是否允许 loopback HTTP 由客户端通过明确的开发模式控制，不能由 URL 参数开启。

### 5.2 商店行为

以下规则适用于具有真实、不可变签名包元数据的已发布条目；任何占位值都不得生成可操作深链接。

1. 只在用户点击“导入”时导航到自定义 URI；
2. 对嵌套 URL 使用标准 `URLSearchParams`/百分号编码，禁止字符串拼接；
3. 页面同时提供包大小、摘要、作者、版本、平台和手动下载入口；
4. 唤起后只显示“已发送到客户端”，除非未来有经过认证的回执机制；
5. 不循环唤起、不后台弹窗，也不尝试探测本机 CDP。

### 5.3 客户端行为

1. 严格解析 scheme 与 action，拒绝重复的必填参数和畸形编码；
2. 显示来源域名、主题 ID、版本、大小并要求确认；
3. 限制重定向次数、最终协议、下载字节数和超时；
4. 将下载保存到新建的私有临时目录，先校验整个包的 SHA-256；
5. 安全解包并逐文件校验清单，拒绝绝对路径、`..`、替代数据流、符号链接和大小超限；
6. 校验 JSON schema、主题字段白名单、图片真实格式、像素尺寸及兼容版本；
7. 原子安装并保留失败回滚点；
8. 成功后再次询问是否立即切换，不把安装自动等同于应用。

URL 中的 `id` 与 `version` 仅用于在下载前展示。它们和包内信息不一致时必须中止，并向用户报告来源不一致。

## 6. 信任与审核模型

### 完整性

- 商店记录不可变版本 URL、字节大小和 SHA-256；
- 发布后改变包内容必须创建新版本，不得只替换同一 URL；
- 客户端以下载到的原始字节计算摘要。

### 发布者身份

- 发布包必须使用受信任的 Ed25519 密钥签名，`keyId` 只作为客户端密钥环索引；
- 签名覆盖 RFC 8785 规范化清单，清单内的 SHA-256 进一步绑定全部主题资源；
- 发布者密钥必须支持轮换，开发模式下的自签名包也不得跳过数学签名验证；
- 密钥撤销与恶意版本撤回需要独立、可缓存的元数据。

### 内容安全

- 自动检查文件类型、解包行为、尺寸、字段白名单和 URL；
- 人工检查预览与实际主题一致性、版权、冒充和不当内容；
- 已发布主题仍可举报、下架或阻止特定版本继续安装；
- 主题许可证必须由作者显式声明，商店代码的 MIT 许可证不会自动覆盖主题作品。

## 7. 版本与兼容性

主题、包 schema、目录 schema 和客户端应各自版本化：

- 主题 `version` 使用 SemVer；
- `theme.json.packageVersion` 控制包契约，`theme.json.schemaVersion` 控制运行时主题字段；
- 目录 `schemaVersion` 控制商店元数据读取；
- `engineVersion` 与 `platforms` 决定客户端最终校验，商店目录可保留对应字段用于提前提示。

商店可以隐藏明显不兼容的导入按钮，但客户端永远是最终裁决者。旧客户端不认识新 schema 时必须安全失败。

## 8. 演进顺序

1. 发布已签名的端到端样例包，并为目录 schema、包 Schema 和安全限制补齐 CI 校验；
2. 在客户端实现本地文件安装，再增加 `dreamskin://` URL 入口；
3. 用端到端样例验证商店链接、下载、哈希错误、兼容错误和回滚；
4. 上线密钥轮换、撤回清单与 Windows 对齐；
5. 确有在线写入需求后，再加入身份、D1、R2、上传和审核队列。

这条顺序把协议和安全边界放在平台规模化之前，也让静态 MVP 能持续作为故障时的只读目录。
